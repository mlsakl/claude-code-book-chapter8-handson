# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**TaskCLI** - Gitと一体化した開発者向けタスク管理CLIツール

### プロダクトコンセプト
- **ターミナル完結**: GUIツールに切り替えることなく、ターミナル上で全てのタスク管理を完結させる
- **Git統合**: タスクとGitブランチ・コミット・PRを自動的に紐付け、開発フローをシームレスに連携する
- **シンプル高速**: 最小限のコマンドで即座に操作できる、学習コストの低い設計

### プロダクトビジョン
開発者がターミナルから離れることなく、効率的にタスクを管理できるCLIツールを提供する。
コードを書くこととタスクを管理することを自然につなげ、コンテキストスイッチによる集中力の分断をゼロにする。
GitのブランチやコミットとタスクIDを自動で紐付けることで、「今何をやっているか」が常に明確な状態を維持する。
JiraやAsanaのような重厚なツールではなく、開発者が本当に必要な機能に特化した軽量なツールを目指す。

### 目的
- ターミナルとGUIツールの切り替えによるコンテキストスイッチを排除する
- タスクとGitブランチ・コミット・PRの手動での紐付け作業を自動化する
- 個人開発者から小規模チームまでが、1分で導入して即座に使い始められるツールを提供する

## ターゲットユーザー

### プライマリーペルソナ: 山田健二（28歳、フリーランスエンジニア）
- フリーランスで複数プロジェクトを並行して担当（月3〜5件）
- ターミナル中心の開発環境（VSCode + 統合ターミナル、または純粋なターミナル）
- GitHub Issuesでタスク管理を試みているが、ローカルの作業との連携が取れていない
- **課題**: 画面を切り替えるたびに集中が途切れる。今どのタスクのためにどのブランチで作業しているか分からなくなる
- **期待する解決策**: ターミナルだけで完結するタスク管理。Gitブランチが自動で紐付く
- **1日のワークフロー**: ターミナルを開く → 今日のタスクを確認 → タスクを開始（ブランチ自動作成） → コーディング → タスク完了（PR自動作成）

### セカンダリーペルソナ: 鈴木美咲（35歳、スタートアップのテックリード）
- 2〜5人の小規模チームのリードエンジニア
- チームのタスク状況をリアルタイムで把握したい
- JiraやAsanaは機能が多すぎてチームへの導入コストが高い
- **課題**: 朝会でチームの進捗確認に時間がかかる。GitHubとタスクの状態が乖離しやすい
- **期待する解決策**: CLIでさっとチーム全体の状況確認。GitHub連携で自動的に進捗更新
- **1日のワークフロー**: 朝会前に `task list --team` で全員の状況確認 → 作業分配 → 夕方に進捗確認

## 機能要件

### コア機能（MVP）

#### 1. タスクの基本CRUD操作

**ユーザーストーリー**:
開発者として、ターミナルから離れずにタスクを管理するために、コマンドラインでタスクの作成・一覧・詳細確認・完了・削除ができる機能が欲しい

**受け入れ条件**:
- [ ] `task add "タスク名"` でタスクを作成でき、一意のIDが自動採番される
- [ ] `task list` で全タスクをID・ステータス・タイトル・関連ブランチ付きで一覧表示できる
- [ ] `task show <id>` でタスクの詳細（作成日時、ステータス履歴、関連コミット）を確認できる
- [ ] `task done <id>` でタスクをcompleted状態に変更できる
- [ ] `task delete <id>` で確認プロンプト付きでタスクを削除できる
- [ ] 全コマンドが100ms以内に応答する

**優先度**: P0（必須）

---

#### 2. タスクステータス管理

**ユーザーストーリー**:
開発者として、タスクの進捗状態を把握するために、4段階のステータス管理機能が欲しい

**受け入れ条件**:
- [ ] `open`（新規）、`in_progress`（作業中）、`completed`（完了）、`archived`（アーカイブ）の4ステータスをサポートする
- [ ] `task status <id> <status>` でステータスを手動変更できる
- [ ] ステータス変更履歴が記録され、`task show <id>` で確認できる
- [ ] `task list` でステータスが色分けして表示される

**優先度**: P0（必須）

---

#### 3. GitブランチとTaskの自動連携

**ユーザーストーリー**:
開発者として、タスクとコードの乖離をなくすために、タスクを開始したときに自動でGitブランチが作成・切り替えされる機能が欲しい

**受け入れ条件**:
- [ ] `task start <id>` 実行時に `feature/task-<id>-<タイトルのスラッグ>` 形式のブランチが自動作成・切り替えされる
- [ ] タスクのステータスが自動的に `in_progress` に変更される
- [ ] Gitリポジトリ外での `task start` は適切なエラーメッセージを表示する
- [ ] 既存ブランチと名前が衝突する場合、ユーザーに確認を求める
- [ ] `task list` でタスクと関連ブランチが対応表示される

**優先度**: P0（必須）

---

#### 4. データの永続化（ローカルJSONストレージ）

**ユーザーストーリー**:
開発者として、タスクデータを安全に保存するために、ローカルファイルにタスクデータが永続化される機能が欲しい

**受け入れ条件**:
- [ ] `.task/tasks.json` にタスクデータが保存される
- [ ] プロセス終了後もデータが保持される
- [ ] JSONファイルはGitで管理可能な形式（チームで共有可能）
- [ ] ファイルが破損した場合、適切なエラーメッセージを表示しデータ損失を最小化する
- [ ] `task backup` コマンドで任意のバックアップを作成できる

**優先度**: P0（必須）

---

#### 5. CLIインターフェース全体設計

**ユーザーストーリー**:
開発者として、直感的にツールを使い始めるために、明確なヘルプと分かりやすいエラーメッセージが欲しい

**受け入れ条件**:
- [ ] `task --help` および `task <command> --help` で各コマンドの使い方を確認できる
- [ ] 存在しないコマンドやIDを指定した場合、分かりやすいエラーメッセージが表示される
- [ ] 初めて使うユーザーが5分以内に基本操作（add/list/start/done）を習得できる
- [ ] `task init` でプロジェクトへの初期設定が完了する

**優先度**: P0（必須）

---

### CLIインターフェース仕様（MVP）

```bash
# 初期設定
task init                           # カレントディレクトリでTaskCLIを初期化

# タスク操作
task add "タスク名"                  # タスク作成
task add "タスク名" --priority high  # 優先度付きタスク作成
task list                           # 全タスク一覧
task list --status in_progress      # ステータスでフィルタ
task show 1                         # タスク詳細表示
task start 1                        # タスク開始（ブランチ自動作成）
task done 1                         # タスク完了
task delete 1                       # タスク削除（確認プロンプト付き）
task status 1 archived              # ステータス手動変更

# バックアップ
task backup                         # バックアップ作成

# ヘルプ
task --help                         # 全コマンド一覧
task add --help                     # コマンド個別ヘルプ
```

### 重要機能（初期リリース後）

#### 6. GitHub Issues連携

**ユーザーストーリー**:
開発者として、GitHub Issuesとの重複管理をなくすために、ローカルタスクとGitHub Issuesを同期できる機能が欲しい

**受け入れ条件**:
- [ ] `task sync` でローカルタスクとGitHub Issuesを双方向同期できる
- [ ] `task import --github` でGitHub IssuesをローカルタスクにインポートできるGitHub Personal Access Tokenを使用した認証
- [ ] タスク完了時にGitHub Issueを自動クローズするオプションがある

**優先度**: P1（重要）

---

#### 7. タスク完了時の自動処理

**ユーザーストーリー**:
開発者として、タスク完了後の手作業を削減するために、`task done` 実行時に関連するGit操作を自動化できる機能が欲しい

**受け入れ条件**:
- [ ] `task done <id>` でブランチのmainへのマージをオプション実行できる
- [ ] `task done <id> --pr` でGitHub PRを自動作成できる
- [ ] 各自動処理の前に確認プロンプトが表示される（スキップオプションあり）

**優先度**: P1（重要）

---

#### 8. タスクの絞り込みと検索

**ユーザーストーリー**:
開発者として、多数のタスクから必要なものを素早く見つけるために、フィルタリングと全文検索機能が欲しい

**受け入れ条件**:
- [ ] `task list --status <status>` でステータスフィルタが機能する
- [ ] `task search "キーワード"` でタスクタイトル・説明の全文検索ができる
- [ ] `task list --sort priority` で優先度順ソートができる
- [ ] 1000件のタスクでも1秒以内に結果を返す

**優先度**: P1（重要）

---

#### 9. 優先度と期限の管理

**ユーザーストーリー**:
開発者として、重要なタスクを見落とさないために、優先度と期限を設定・管理できる機能が欲しい

**受け入れ条件**:
- [ ] `--priority <high|medium|low>` オプションで優先度設定ができる
- [ ] `--due <YYYY-MM-DD>` オプションで期限設定ができる
- [ ] 期限が近いタスク（3日以内）が`task list`で強調表示される
- [ ] 期限超過タスクに警告表示がある

**優先度**: P1（重要）

## 非機能要件

### パフォーマンス
- コマンド実行時間: 100ms以内（平均的なPC環境、GitHub API呼び出しを除く）
- タスク一覧表示: 1000件まで1秒以内
- GitHub API連携コマンド: 5秒以内（タイムアウト表示付き）

### ユーザビリティ
- 新規ユーザーが `task --help` を読んで5分以内に基本操作（add/list/start/done）を習得できる
- エラーメッセージは問題の原因と解決方法を日本語で明示する
- 破壊的操作（delete、merge等）には必ず確認プロンプトを表示する

### 信頼性
- タスクデータの損失ゼロ（書き込み前に自動バックアップ）
- ファイル書き込みはアトミック操作で実施し、中断による破損を防ぐ
- Gitコマンド失敗時はタスクのステータス変更をロールバックする

### セキュリティ
- GitHub Personal Access Tokenは環境変数または設定ファイル（パーミッション600）で管理し、コード・タスクデータには含めない
- `.task/config.json` はデフォルトで `.gitignore` に追加する（トークン保護）

### スケーラビリティ
- タスク数1万件までJSONファイルで対応し、超過時はSQLite移行を検討する
- クロスプラットフォーム対応: macOS、Linux、Windows（Git Bash）

### 互換性
- Node.js v18以上で動作する
- npm経由でグローバルインストール可能（`npm install -g taskcli`）

## スコープ外

明示的にスコープ外とする項目（MVP〜P1フェーズ）:

- Webブラウザによる操作インターフェース
- モバイルアプリ
- クラウド専用の同期サービス（Git経由の同期は対象内）
- GitLab・Bitbucketとの連携（GitHub優先）
- プラグインシステム・カスタムコマンド定義
- Slackやその他外部サービスとの通知連携
- タイムトラッキングの詳細分析・レポート機能
